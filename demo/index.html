Skills • Agents • Serve API • Knowledge Base • Containerlab

The Problem

Configuring network devices requires:

  • Deep knowledge of vendor-specific CLI syntax
  • Reading 1000+ page documentation PDFs
  • Manually writing configs per device
  • Building lab topologies by hand

What if an agent could read all the docs and generate configs for you?

The Solution

Skill 1 OpenAPI Agent

Reads any OpenAPI spec + app rules, calls the right endpoints automatically.

Skill 2 Nokia NetOps

Queries ChromaDB knowledge base of Nokia docs, generates configs + Containerlab topologies.

Both exposed via OpenCode Serve API — callable from n8n, other agents, or CLI.

The 4-Layer Architecture

A2A Agent-to-Agent Protocol
Agents discover and delegate tasks to each other
SERVE OpenCode Serve API
HTTP bridge — makes agents callable from anywhere
DATA ChromaDB / MCP / Datasources
Vector KB, databases, external tools
SKILLS Skills + AGENTS.md
Domain knowledge — what to do and how

Nokia NetOps Skill

Generates device configs from a knowledge base of Nokia documentation


---
name: nokia-netops
description: "Generate Nokia SR Linux and SR OS device
  configurations and Containerlab topologies by querying
  a ChromaDB knowledge base built from official Nokia
  documentation."
---

Press down for details

What It Can Do

  • Single device: "Configure hostname and cards on SR OS"
  • Multi-device: "Configure 5 SROS routers with eBGP full mesh"
  • Mixed platforms: "2 SROS spines + 3 SR Linux leaves"
  • Full lab: "Generate configs + Containerlab topology"
  • Specific features: "Add OSPF area 0 with BFD on all links"

Skill Steps

  1. Identify platform — SR Linux, SR OS, or both
  2. Query ChromaDB for relevant config syntax
  3. Determine device count and roles (spine/leaf/PE/CE)
  4. Generate configs with correct CLI syntax
  5. Assign addressing — loopbacks, P2P links, management
  6. Generate Containerlab .clab.yml if requested
  7. Output all files + summary table

Ingestion Script


# Install dependencies
pip install -r ingest/requirements.txt

# Run ingestion
python3 ingest/ingest.py

# What happens:
# 1. Reads sources.yaml (URLs to Nokia docs)
# 2. Fetches each HTML page / PDF
# 3. Strips nav/header/footer
# 4. Chunks text (1000 chars, 200 overlap)
# 5. Stores in ChromaDB with metadata
    

# Query the knowledge base
python3 scripts/query-kb.py \
  --query "BGP neighbor configuration SR Linux" \
  --platform srlinux \
  --n-results 5

sources.yaml

Calling the Nokia Agent via HTTP


# 1. Create a session
curl -X POST http://localhost:4096/session \
  -H "Content-Type: application/json" \
  -d '{"title": "nokia-lab"}'
# → { "data": { "id": "ses_abc123" } }
    

# 2. Send config request to nokia-netops-agent
curl -X POST http://localhost:4096/session/ses_abc123/message \
  -H "Content-Type: application/json" \
  -d '{
    "agent": "nokia-netops-agent",
    "parts": [{
      "type": "text",
      "text": "Configure 5 SROS routers with eBGP full mesh. Generate all configs and a containerlab topology."
    }]
  }'

A2A — Agent-to-Agent Protocol

Google's open protocol (Linux Foundation) for agent interoperability 

  Orchestrator Agent              Nokia NetOps Agent
       │                              │
       │  1. Read Agent Card          │
       │─────────────────────────────>│  (discover capabilities)
       │                              │
       │  2. SendMessage              │
       │─────────────────────────────>│  "5 SROS routers, eBGP"
       │                              │
       │  3. Task {WORKING}           │
       │<─────────────────────────────│
       │                              │  (queries KB, generates configs)
       │  4. Task {COMPLETED}         │
       │<─────────────────────────────│
       │     + artifacts              │  (configs + topology files)
