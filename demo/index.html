Agentic Skills Platform

OpenCode + Nokia NetOps + ChromaDB

Skills • Agents • Serve API • Knowledge Base • DTaaS • Testing

The Problem

Configuring network devices requires:

  • Deep knowledge of vendor-specific CLI syntax
  • Reading 1000+ page documentation PDFs
  • Manually writing configs per device
  • Building lab topologies by hand

What if an agent could read all the docs and generate configs for you?

The Solution — 4 Skills

1 OpenAPI Agent

Reads any OpenAPI spec + app rules, calls endpoints automatically.

2 Nokia NetOps

Queries ChromaDB, generates device configs + Containerlab topologies.

3 DTaaS Deploy

Deploys generated labs to the Digital Twin as a Service platform via API.

4 Test Writer

Generates and runs network test cases to validate deployed labs.

All exposed via OpenCode Serve API — callable from Azure AI Foundry, Glean, other agents, or CLI.

The 4-Layer Architecture

A2A Agent-to-Agent Protocol
Agents discover and delegate tasks to each other
SERVE OpenCode Serve API
HTTP bridge — makes agents callable from anywhere
DATA ChromaDB / MCP / Datasources
Vector KB, databases, external tools
SKILLS Skills + AGENTS.md
Domain knowledge — what to do and how

What Is a Skill?

A skill is a markdown file that teaches an AI agent how to do a specific job

Think of it like...

A runbook for a new team member:
"Here's how we do X, step by step,
with these rules and these tools."
Not code — instructions

No Python, no JavaScript.
Just structured text that tells the
LLM what to do and how.

Skill File Structure


# .opencode/skills/my-skill/SKILL.md

---
name: my-skill

You are a [role]. Your job is to [task].

## Steps
1. First, do this
2. Then check that
3. Finally, output this

## Rules
- Always validate before acting
- Never do X without confirmation
- Use Y format for output

YAML frontmatter = identity • Markdown body = the actual instructions

Nokia NetOps Skill

Generates device configs from a knowledge base of Nokia documentation


---
name: nokia-netops
description: "Generate Nokia SR Linux and SR OS device
  configurations and Containerlab topologies by querying
  a ChromaDB knowledge base built from official Nokia
  documentation."
---

Press down for details

What It Can Do

  • Single device: "Configure hostname and cards on SR OS"
  • Multi-device: "Configure 5 SROS routers with eBGP full mesh"
  • Mixed platforms: "2 SROS spines + 3 SR Linux leaves"
  • Full lab: "Generate configs + Containerlab topology"
  • Specific features: "Add OSPF area 0 with BFD on all links"

Skill Steps

  1. Identify platform — SR Linux, SR OS, or both
  2. Query ChromaDB for relevant config syntax
  3. Determine device count and roles (spine/leaf/PE/CE)
  4. Generate configs with correct CLI syntax
  5. Assign addressing — loopbacks, P2P links, management
  6. Generate Containerlab .clab.yml if requested
  7. Output all files + summary table

How Foundry Connects

  • OpenAPI import: Import specs/opencode-api.json as a Foundry tool — agents call our endpoints natively
  • MCP server: Register OpenCode as an MCP tool provider — Foundry agents discover skills automatically
  • A2A protocol: Publish an Agent Card — Foundry's multi-agent orchestrator delegates tasks via A2A
  • Logic Apps: HTTP connector calls POST /session/{id}/message as a workflow step

Why Container Apps / AKS?

  • Scale to zero — Container Apps only charges when agents are processing requests
  • VNET integration — OpenCode reaches DTaaS API (10.0.0.12) and LLM backends over private network
  • Managed identity — no API keys in env vars; Azure OpenAI auth via Entra ID
  • Ingress + TLS — built-in HTTPS endpoint for Foundry / Glean to call
  • AKS for production — persistent volumes for ChromaDB, network policies, multi-replica for HA

Connect from Your Shell

Use your local opencode CLI with the running server


# Full TUI — reuses server's skills, agents, ChromaDB
opencode --attach http://localhost:4096

# One-off command
opencode run --attach http://localhost:4096 \
  "Configure 3 SROS routers with eBGP"

# Or symlink skills globally (no server needed)
ln -s ~/poc/skills/.opencode/skills/* ~/.config/opencode/skills/
ln -s ~/poc/skills/.opencode/agents/* ~/.config/opencode/agents/
--attach is the cleanest — your shell talks to the container,
which already has skills, ChromaDB, and LLM provider configured. No duplication.
